## 1. 产品定位与核心价值

### 1.1 一句话定位
> "每天一杯咖啡的时间，把我关心的每个领域里**昨天真正发生的事**讲清楚，没有公关空话。"

### 1.2 核心价值（与普通新闻 App 的区别）
1. **垂直 + 自定义**：分类、关键词、提示词、输出格式都由用户自己定义，不是平台推送什么看什么。
2. **强时效 + 反噪音**：每条新闻必须是**目标日期当天**真实发生的具体事件，过滤通稿、空话、过期内容。
3. **AI 二次提炼**：云端开源大模型把多源搜索结果去重、筛选、结构化，输出"干货简报"而非链接列表。
4. **可追问**：对任意一条简报可继续对话深挖（基于已检索素材的二次问答）。

### 1.3 目标用户
对特定领域（科技 / 投资 / 行业研究 / 个人兴趣）有**每日跟踪需求**的专业用户与重度信息消费者。

## 2. 技术栈选型

设计原则：**业务逻辑尽量上云，客户端只做配置 + 渲染**。手机端不直接跑多轮搜索和长文本推理（费流量、费电、慢、Key 不安全）。

### 2.1 客户端
| 模块 | 选型 | 理由 |
| --- | --- | --- |
| UI 框架 | **Jetpack Compose** | 声明式 UI，`WindowSizeClass` 天然适配手机/平板 |
| 架构 | **MVVM + Clean Architecture**（UI / Domain / Data 三层） | 业务层与平台解耦，为 iOS 迁移留好边界 |
| 跨平台核心（可选） | **Kotlin Multiplatform (KMP)** | 网络/数据/领域层一套代码，未来 iOS 复用 |
| 网络 | **Ktor Client**（KMP）或 Retrofit + OkHttp（纯 Android） | 异步、拦截器、易测试 |
| 本地存储 | **Room**（纯 Android）/ **SQLDelight**（KMP） | 离线缓存简报、配置 |
| 异步 | Kotlin Coroutines + Flow | 标准方案 |
| Markdown 渲染 | **compose-markdown**（如 `com.mikepenz:multiplatform-markdown-renderer`） | 简报是 Markdown，需结构化渲染 |
| 依赖注入 | Hilt（Android）/ Koin（KMP） | — |
| 图片 | Coil | — |

> **iOS 迁移策略**：MVP 阶段用纯 Android 也可以，但**领域层和数据层的接口要写成平台无关**（不引用 Android Framework）。第三阶段要么用 Compose Multiplatform 共享 UI，要么用 SwiftUI 写壳、复用 KMP 业务层。**现在就要约束好分层，避免未来重构。**

### 2.2 后端
| 模块 | 选型 | 理由 |
| --- | --- | --- |
| 框架 | **Python + FastAPI** | 生态最适合搜索/LLM 编排，异步性能足够 |
| 任务调度 | **APScheduler**（MVP）→ **Celery + Redis**（规模化） | 每日定时生成简报 |
| 缓存/队列 | **Redis** | 缓存简报、限流、去重 |
| 数据库 | **PostgreSQL** | 存配置、简报、来源、用户 |
| 部署 | Docker + 任一云主机；或 Serverless（云函数 + 定时触发器） | MVP 一台 2C4G 足够 |

### 2.3 搜索源（**整个项目最关键的一环，见 §4.2 / §8.1**）
| 方案 | 产品 | 特点 | 建议 |
| --- | --- | --- | --- |
| AI 搜索 API（首选） | **Tavily** / **Exa** | 为 LLM 优化，原生支持 `days` / 时间范围过滤、返回正文摘要 | **MVP 主选**：省去自己爬正文 |
| 通用搜索 API | **Brave Search API** / **SerpAPI（Google News）** / **Bing News（迁移中需替代）** | 结果全、可指定 `freshness`/时间 | 作为补充源、交叉验证 |
| 新闻聚合 API | **GNews** / **NewsAPI** / **NewsData.io** | 直接给结构化新闻 + 发布时间 | 中文覆盖一般，作辅助 |
| RSS 聚合 | 特定媒体 RSS（36氪、机器之心、官方博客等） | 免费、时效高、来源可信 | 针对每个分类维护一份白名单 RSS |

> **结论**：MVP 用 **Tavily（主）+ 分类专属 RSS 白名单（辅）** 组合，既保证时效与正文质量，又控制成本。

### 2.4 云端开源大模型
| 用途 | 模型 | 接入渠道 |
| --- | --- | --- |
| 主力提炼 | **DeepSeek-V3 / deepseek-chat** | DeepSeek 官方 API（性价比高、中文强） |
| 备选/多源 | **Qwen2.5-72B-Instruct**、**GLM-4** | **硅基流动 SiliconFlow** / **OpenRouter**（一个 Key 多模型，便于切换比价） |

> 通过 **OpenRouter / SiliconFlow** 这类聚合网关接入，可以用一套 OpenAI 兼容协议随时切换底层开源模型，不被单一供应商绑定。

### 3.1 每日简报生成流程（核心业务流）
```
[定时任务触发 (每日 06:00) 或 用户手动刷新]
      │
      ▼
[取该分类配置：关键词列表 + 自定义 Prompt + 目标日期(默认 T-1)]
      │
      ▼
[搜索适配层：对每个关键词调用 Tavily(限定 days=1) + 拉取白名单 RSS]
      │
      ▼
[清洗预处理：① 解析发布时间 ② 强制剔除≠目标日期 ③ URL/标题去重 ④ 按相关度排序，取 Top 20–30]
      │
      ▼
[构建 Prompt：系统提示词(规则) + 用户自定义格式 + 结构化素材(JSON)]
      │
      ▼
[调用云端开源大模型 → 输出结构化 Markdown 简报]
      │
      ▼
[落库(BriefingReport) + 写缓存 + 记录引用来源]
      │
      ▼
[客户端拉取并渲染；本地缓存供离线查看]
```

**关键决策**：用户"早上打开 App"看到的是**已经由定时任务在凌晨生成好的简报**（秒开），而不是当场等待几十秒搜索+推理。手动刷新作为补充。

## 4. 后端服务详细设计

### 4.1 模块划分
- `api/`：REST 接口、鉴权、限流、参数校验。
- `orchestrator/`：简报生成主流程（串起搜索→清洗→提炼→落库）。
- `search/`：搜索适配层，定义统一 `SearchProvider` 接口，下挂 Tavily / Brave / RSS 等实现。
- `refine/`：LLM 提炼层，封装 OpenAI 兼容协议，统一重试/超时/降级。
- `scheduler/`：定时任务。
- `repository/`：数据访问。

### 4.2 搜索适配层（统一接口）
```python
class NewsItem(BaseModel):
    title: str
    url: str
    source: str            # 媒体名
    published_at: datetime # 解析后的发布时间（关键字段）
    snippet: str           # 正文摘要
    raw_text: str | None   # 可选全文

class SearchProvider(Protocol):
    async def search(self, query: str, target_date: date, max_results: int) -> list[NewsItem]: ...

# 实现：TavilyProvider / BraveProvider / RssProvider
# Orchestrator 并发调用多个 Provider，合并结果后统一清洗
```

**日期过滤三道防线**（对应需求里"必须是目标日期当天"）：
1. **检索层**：Tavily 用 `days=1` / Brave 用 `freshness`，把搜索时间窗压到最近一天。
2. **清洗层**：解析每条 `published_at`，**硬过滤**掉 `published_at.date() != target_date` 的条目（代码层面剔除，不依赖模型自觉）。
3. **提炼层**：Prompt 再次强调"非目标日期一律跳过、无新闻整块省略、禁止编造"。

> 这是 Gemini 文档没讲透的核心：**日期准确性必须在代码层兜底，不能只靠 Prompt。** 搜索引擎返回的日期常有误，需以多源交叉 + 正则/解析为准。

### 4.3 LLM 提炼层
- 统一走 **OpenAI 兼容 `/chat/completions`**，通过配置切换 `deepseek-chat` / `Qwen` / `GLM`。
- 输入：系统提示词（规则，见 §7） + 用户自定义格式 + **结构化素材 JSON**（标题/来源/日期/摘要）。
- 输出：Markdown 简报字符串 + 本次引用的来源列表（用于落库与"可追问"）。
- 健壮性：超时(60s)、失败重试(2 次)、模型降级（主模型不可用切备用网关）、Token 预算控制（素材先截断到 Top N）。

### 4.4 REST 接口契约（MVP）

> 鉴权：MVP 用简单的 `Authorization: Bearer <app_token>`；正式上架前接入账号体系（见 §9）。

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/v1/categories` | 获取用户的分类列表 |
| POST | `/api/v1/categories` | 新建分类（名称/关键词/Prompt/启用） |
| PUT | `/api/v1/categories/{id}` | 修改分类配置 |
| DELETE | `/api/v1/categories/{id}` | 删除分类 |
| GET | `/api/v1/categories/{id}/briefing?date=YYYY-MM-DD` | 获取某分类某日简报（默认最新） |
| POST | `/api/v1/categories/{id}/refresh` | 手动触发该分类生成（异步，返回 task_id） |
| GET | `/api/v1/tasks/{task_id}` | 查询生成任务状态/进度 |
| POST | `/api/v1/briefings/{id}/ask` | 基于该简报素材追问（对话） |

**简报响应示例**
```json
{
  "id": "brf_20260602_ai",
  "category_id": "cat_ai",
  "report_date": "2026-06-02",
  "generated_at": "2026-06-03T06:02:11Z",
  "status": "ready",          // pending | running | ready | empty | failed
  "content_markdown": "════...📅 AI 大模型动态简报...",
  "sources": [
    {"title": "...", "media": "机器之心", "url": "https://...", "published_at": "2026-06-02"}
  ],
  "model": "deepseek-chat",
  "token_usage": {"prompt": 5123, "completion": 1840}
}
```
> `status = "empty"` 表示当日该分类确实无符合条件的新闻——客户端要友好展示"今日该领域无重大动态"，而不是报错。

## 5. 数据模型

### 5.1 云端（PostgreSQL）
**users**（MVP 可先用单设备匿名 token，上架前补全）
- `id`, `device_id`, `created_at`

**category_config**
- `id` (PK)
- `user_id` (FK)
- `name`：分类名，如 "AI 大模型"
- `search_keywords`：JSON 数组，如 `["DeepSeek 深度求索", "Anthropic Claude", ...]`
- `rss_whitelist`：JSON 数组，分类专属 RSS（可空）
- `custom_prompt`：用户自定义的筛选与输出格式
- `target_offset_days`：目标日期偏移，默认 `1`（即 T-1）
- `schedule_cron`：生成时间，默认每日 `0 6 * * *`
- `is_enabled` (bool)
- `sort_order` (int)
- `created_at`, `updated_at`

**briefing_report**
- `id` (PK)
- `category_id` (FK)
- `report_date`：目标日期
- `generated_at`
- `status`：pending/running/ready/empty/failed
- `content_markdown`
- `model`、`token_prompt`、`token_completion`
- `error`（失败原因，可空）
- 唯一约束：`(category_id, report_date)`

**report_source**（用于"可追问"和可信度展示）
- `id`, `report_id` (FK), `title`, `media`, `url`, `published_at`, `snippet`

### 5.2 客户端（Room / SQLDelight）
基本镜像云端结构，用于**离线查看 + 秒开**：
- `CategoryConfigEntity`（同步自云端，支持本地编辑后上行）
- `BriefingReportEntity`（缓存最近 N 天简报）
- `SourceEntity`
- `SyncMeta`（记录最后同步时间、待上行变更）

## 6. 客户端设计（手机 / 平板自适应）

### 6.1 三大页面
1. **资讯看板 Dashboard**：卡片/瀑布流展示各启用分类的"今日简报"入口（标题 + 摘要 + 生成时间 + 状态徽标）。
2. **订阅配置**：分类的增删改查；编辑关键词、自定义 Prompt、目标日期偏移、生成时间、选用模型；提供"预览/调试"按钮即时生成一次看效果。
3. **简报详情 + 追问**：Markdown 结构化渲染；底部对话框可基于本篇素材继续追问；支持分享/复制/收藏。

### 6.2 响应式布局（核心适配点）
用 `WindowSizeClass` 区分：
- **Compact（手机竖屏）**：底部导航栏（看板 / 配置 / 我的）+ 顶部分类 Tabs；详情为整屏。
- **Medium/Expanded（平板/横屏）**：
  - 用 `NavigationSuiteScaffold` 把底部导航变为**左侧侧边栏**；
  - 简报采用 **List-Detail 双栏**：左列分类/简报列表，右列直接显示选中简报，充分利用大屏。

### 6.3 关键交互
- **下拉刷新**：触发 `POST /refresh` → 轮询 `task` → 完成后更新。
- **空态**：`status=empty` 显示"📭 今日该领域暂无符合条件的新闻"。
- **离线**：无网时读本地缓存，并提示"显示的是 X 月 X 日缓存"。
- **错误**：生成失败显示重试入口与简要原因。

## 7. Prompt 工程与质量控制

### 7.1 分层 Prompt 设计
后端把"规则"放在 **System Prompt（平台统一、不可被用户覆盖的硬约束）**，把"领域与输出格式"放在 **User Prompt（用户自定义）**，再把素材作为结构化数据传入。这样既尊重用户的自定义格式，又能保证日期/反编造等底线规则不被破坏。

**System Prompt（平台硬约束，模板）**
```
你是一个严谨的每日资讯提炼助手。你只能基于【素材】中提供的新闻进行提炼，禁止使用素材外的任何信息，禁止编造。

时间规则：
- 今日日期：{{current_date}}
- 目标日期：{{target_date}}
- 只保留发布日期 = 目标日期 的新闻；其余一律丢弃。
- 某板块无符合条件的新闻：整块省略，不写任何占位文字。

内容规则：
- 每条必须是目标日期当天真实发生的具体事件，不得泛泛概括。
- 每条必须注明：来源媒体 + 发布日期（必须是目标日期）。
- 无法确认来源或日期的信息，直接跳过。
- 禁止"持续发力""整体趋势"等空话。

输出规则：
- 严格按【用户输出格式】渲染；用户未指定时，用默认简报格式。
- 全程中文。
```

**User Prompt（用户自定义，举例为用户给的 AI 大模型格式）**
```
领域：AI 大模型
关注对象：DeepSeek / Claude / Gemini / ChatGPT / Grok / Kimi / GLM / MiniMax / NotebookLM·Jules
输出格式：{{用户粘贴的 📅 简报模板}}
```

**素材（程序注入，非自然语言）**
```json
{ "items": [ {"title":"...","media":"...","published_at":"2026-06-02","snippet":"..."} , ... ] }
```

### 7.2 质量控制
- **代码层兜底日期**（§4.2 三道防线），不把日期准确性全压给模型。
- **来源核对**：模型输出的每条来源必须能在 `report_source` 中找到对应项，否则标红/丢弃（后处理校验）。
- **可重放**：保存每次的素材快照 + Prompt + 模型版本，便于复现和调参。
- **A/B Prompt**：把 System Prompt 做成可版本化配置，便于持续优化。

## 8. 难点与解决方案

| # | 难点 | 解决方案 |
| --- | --- | --- |
| 8.1 | **新闻时效与日期噪音**（最大难点）：搜索结果发布日期常不准/缺失 | ① 优先用支持时间窗的 AI 搜索 API；② 多源交叉，以能解析到的真实 `published_at` 为准；③ 代码硬过滤非目标日期；④ 缺日期的条目默认丢弃而非保留 |
| 8.2 | **大模型 Token 成本与上下文** | 先对搜索结果做去重 + 相关度排序，仅取 Top 20–30 条"标题+摘要"喂给模型（RAG 思想），不喂全文 |
| 8.3 | **响应速度**（用户不想等几十秒） | 定时任务凌晨预生成，App 打开秒开缓存；手动刷新才实时生成并显示进度 |
| 8.4 | **搜索/LLM 成本随分类增长** | 相同关键词跨用户共享缓存；按分类每日只生成一次；设置每用户每日刷新配额 |
| 8.5 | **API Key 安全** | 所有搜索/LLM Key 只存后端环境变量/密管，**绝不下发到客户端**；客户端只持 App Token |
| 8.6 | **中文新闻覆盖** | 国际 API 中文覆盖有限 → 为每个分类维护**中文媒体 RSS 白名单**（如机器之心、36氪、官方公众号转 RSS、车企/政务官网） |
| 8.7 | **iOS 未来迁移成本** | 现在就强制分层、领域层平台无关；优先 KMP，至少保证业务逻辑可移植 |
| 8.8 | **供应商绑定** | LLM 走 OpenAI 兼容网关（OpenRouter/SiliconFlow），可随时切换开源模型比价 |

## 9. 成本、合规与上架

### 9.1 成本量级（粗估，以便预算，非承诺）
- 搜索：Tavily 等按次计费，单分类单日约 8–10 次搜索；多分类 × 每日 → 是主要可变成本，靠缓存与配额控制。
- LLM：DeepSeek/开源模型单价低，单篇简报输入约 5–8K token、输出 1–2K token，成本可控。
- 服务器：MVP 单台 2C4G + 托管 PG/Redis 即可起步。
> 建议上线 **用量看板**（每日搜索次数、token、失败率）便于盯成本。

### 9.2 合规（上架必备）
- **版权与来源**：简报注明来源媒体与链接，定位"摘要 + 引用"，不全文转载；提供原文跳转。
- **AI 生成标识**：按各地法规对 AI 生成内容做显著标识（中国《生成式 AI 服务管理办法》、应用商店要求）。
- **隐私政策 + 用户协议**：收集设备标识/订阅配置需明示；准备隐私清单（iOS）、数据安全表单（Google Play）。
- **内容安全**：政治热点等分类需有内容审核/敏感词策略，尤其面向国内应用商店。
- **资质**：国内上架可能涉及 ICP/相关备案与资质，提前评估。

### 9.3 上架物料
应用图标、启动页、应用商店截图（手机 + 平板）、隐私政策页 URL、分级问卷、版本说明。

## 10. 分阶段开发计划

### 阶段一：MVP（Android 手机 + 核心链路）
**目标**：单分类（先做"AI 大模型"），打开 App 能看到 T-1 的真实、带来源、限定日期的 Markdown 简报。
- 后端：FastAPI 跑通 `搜索(Tavily+RSS) → 清洗(日期硬过滤) → DeepSeek 提炼 → 落库 → 接口`；APScheduler 每日定时生成。
- 客户端：Compose 实现看板 + 简报详情（Markdown 渲染）+ 下拉刷新；本地缓存。
- 配置：先用配置文件/后台预置 1–2 个分类（关键词 + Prompt）。
- **验收标准**：
  1. 简报中每条新闻都标注来源媒体 + 发布日期，且发布日期 = 目标日期；
  2. 无新闻的板块自动省略，全无新闻时显示空态；
  3. 抽查 10 条，来源链接可打开、内容与日期相符，无明显编造；
  4. 冷启动打开已生成简报 < 1.5s。

### 阶段二：自定义 + 多分类 + 平板适配 + 上架准备
- App 内完成分类/关键词/Prompt 的增删改查与"预览调试"。
- 支持用户给的全部分类（新能源车企、体育明星、互联网大厂、国家政治热点…）。
- `WindowSizeClass` 平板双栏 List-Detail 适配。
- "可追问"对话功能。
- 账号体系、隐私政策、AI 标识、商店物料，准备上架 Google Play / 国内商店。

### 阶段三：iOS 跨端
- 复用 KMP 业务/数据层；用 Compose Multiplatform 或 SwiftUI 写 iOS 壳；上架 App Store。
- 推送（每日简报就绪通知）、Widget（桌面小组件看今日要点）。

```
AIHot/
├── docs/
│   └── 开发说明书.md                # 本文档

│   │   ├── scheduler/               # 定时任务
│   │   ├── repository/              # DB 访问
│   │   ├── models/                  # Pydantic / ORM 模型
│   │   ├── prompts/                 # System Prompt 模板(可版本化)
│   │   └── core/                    # 配置、密管、日志
│   ├── tests/
│   ├── pyproject.toml
│   └── Dockerfile
├── android/                         # Android / Compose 客户端
│   └── app/src/main/java/com/infopulse/
│       ├── ui/                      # Compose 页面(看板/配置/详情)
│       ├── domain/                  # 用例、领域模型(平台无关)
│       ├── data/                    # 仓库、网络、Room
│       └── di/

```

**规范要点**
- 后端：类型注解 + Pydantic 校验；所有外部调用（搜索/LLM）统一超时、重试、降级、结构化日志。
- 客户端：UI 层不写业务；网络模型 ≠ 领域模型 ≠ DB 实体，各层有映射。
- 密钥：一律走环境变量 / 密管，仓库内提供 `.env.example`，**严禁提交真实 Key**。
- 文案/格式：简报 Markdown 模板集中维护，便于统一调整。

开发前建议先拍板以下几项（不影响整体架构，但影响选型与排期）：
